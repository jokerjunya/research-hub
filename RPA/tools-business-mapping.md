# RPAツールの特徴と通常業務への適用マッピング（2025/10/04時点）

---

## 🧩 各RPAツールの特徴一覧

| ツール名 | タイプ | 主な特徴 | 向いている業務領域 | 注意点 |
|------------|---------|------------|----------------|-----------|
| **Playwright** | Web操作型 | ・複数ブラウザ対応（Chrome, Edge, Firefox, Safari）  <br>・高速・安定・非同期待機が自動 <br>・テスト・RPA両対応  | Webポータル操作、社内承認、データ収集、スクレイピング | DOM構造変更に弱い。バージョンアップで非互換が起きる場合あり。 |
| **Puppeteer** | Web操作型 | ・Chrome専用で軽量 <br>・導入が容易でNode.jsだけで完結 <br>・Playwrightより学習コスト低い | Chrome中心のWebシステム操作、フォーム自動入力 | Chrome依存。非同期処理・待機制御の明示が必要なケースあり。 |
| **Selenium** | Web操作型 | ・歴史が長く互換性高い <br>・複数言語サポート <br>・テスト自動化で成熟 | レガシーWebシステムの自動化、大規模ブラウザテスト | 比較的遅い。スクリプトが冗長になりやすい。 |
| **Power Automate Desktop (PAD)** | デスクトップ操作型 | ・Windows標準搭載・GUIで操作可能 <br>・Office製品との連携◎ <br>・Citrix／VDI対応 | Excel処理、SAP、Notes、社内アプリ操作 | Windows専用。速度と拡張性に制約。 |
| **pywinauto** | デスクトップ操作型 | ・PythonからWin32/UIAを直接操作 <br>・軽量で柔軟 <br>・細かい制御が可能 | Excel/Notes/SAP等の自動化、内製スクリプト化 | コーディングスキルが必要。UI変更に脆弱。 |
| **UiPath** | 統合型（Web+Desktop+API） | ・GUI開発＋強力なOrchestrator管理 <br>・AI OCR/ドキュメント理解機能 <br>・監査ログや配布も容易 | 大規模運用、文書処理、承認・報告業務 | ライセンス高価。複雑業務で設計が重くなる。 |
| **Automation Anywhere** | 統合型（クラウド中心） | ・ブラウザベースで管理 <br>・AI連携・Bot運用に強い | 大企業向け統合自動化、SaaS連携 | クラウド前提でローカル操作には不向き。 |
| **Make (旧Integromat)** | API連携型 | ・ノーコードでAPIワークフロー構築 <br>・条件分岐・フィルタもGUIで実装可能 | データ転送、Slack通知、レポート生成 | 有料プランでないと実行数制限あり。 |
| **Zapier** | API連携型 | ・UIがシンプル・テンプレート多数 <br>・連携先が豊富（5000以上） | Webサービス連携、メール通知、CRM連携 | 同期遅延・ステップ数制限あり。 |
| **n8n** | API連携型（自社ホスト可） | ・OSSでセルフホスト可能 <br>・プライバシー・コスト両立 <br>・ノーコード＋コード混在対応 | 内製API連携、社内自動連携基盤構築 | 自社サーバ運用の管理負荷あり。 |

## 🏢 通常業務別：向いているツール・手順・注意点

### 1️⃣ 経費申請・承認処理
- **向いているツール**：Playwright / UiPath / Make
- **手順例**：
  1. Web勤怠・経費ポータルへ自動ログイン
  2. 承認待ち一覧を取得
  3. 各行を走査して「交通費あり」条件を判定
  4. 該当行を開いて承認ボタンをクリック
  5. 結果をSlack／メール通知（Make経由）
- **注意点**：
  - DOM構造変更で動作が止まりやすい → 安定セレクタ選定必須
  - 承認操作は冪等性（重複承認）対策が必要

---

### 2️⃣ 月次売上レポート集計
- **向いているツール**：Power Automate Desktop / pywinauto / Make
- **手順例**：
  1. 各部門の売上Excelを取得（フォルダ監視）
  2. 自動で集計・ピボット生成
  3. PDF出力 → Teamsまたはメール送信
- **注意点**：
  - Excelファイル構造変更に弱い
  - フォルダ監視のタイミングとロック制御に注意

---

### 3️⃣ 請求書発行・送付
- **向いているツール**：Make / Zapier / UiPath
- **手順例**：
  1. CRMから受注データ取得（API）
  2. テンプレートに流し込み → PDF生成
  3. ファイル添付メール送信（API）
- **注意点**：
  - APIレート制限に注意
  - ファイル添付サイズ・宛先管理を標準化

---

### 4️⃣ 勤怠承認チェック
- **向いているツール**：Playwright / Puppeteer / UiPath
- **手順例**：
  1. 勤怠ポータルへログイン
  2. 「未承認」リストを取得
  3. 入退館ログや交通費申請の整合チェック
  4. 問題なければ承認
- **注意点**：
  - JS非同期処理に対応した待機処理を設定（Playwright推奨）
  - UI変更時のセレクタメンテナンス

---

### 5️⃣ 顧客データ同期（CRM ↔ 他システム）
- **向いているツール**：Make / n8n / Zapier
- **手順例**：
  1. CRMの更新イベントをWebhookで受信
  2. 対応するシステムへデータ送信（API）
  3. 結果をログ出力
- **注意点**：
  - APIスキーマ変更への耐性を確保
  - エラーハンドリング／再送設計必須

---

### 6️⃣ メール解析・チケット起票
- **向いているツール**：n8n / Make / UiPath
- **手順例**：
  1. Gmail／Outlook APIで新着メール取得
  2. 正規表現またはAIで内容抽出
  3. チケット管理システム（Jira, Backlog 等）へAPI登録
- **注意点**：
  - 個人情報を含む場合は匿名化処理
  - 定期ポーリングよりWebhook連携推奨

---

### 7️⃣ Webスクレイピング・レポート収集
- **向いているツール**：Playwright / Selenium / UiPath
- **手順例**：
  1. Webページへアクセス
  2. ログイン認証処理（Cookie管理）
  3. 必要な表データを取得しCSV保存
- **注意点**：
  - 利用規約違反にならないよう取得範囲確認
  - ページ読み込み待機の最適化が鍵

---

### 8️⃣ 紙文書スキャン→OCR登録
- **向いているツール**：UiPath / Power Automate Desktop / Make
- **手順例**：
  1. スキャナからPDF化
  2. OCR変換（AI Document Understandingなど）
  3. 結果をシステム登録（APIまたはUI操作）
- **注意点**：
  - OCR誤認識を想定し再確認プロセスを設ける
  - センシティブ情報の扱いに注意

---

### 9️⃣ 定期レポート送信
- **向いているツール**：Make / Power Automate Cloud / Zapier
- **手順例**：
  1. スケジュールトリガーで実行
  2. 各データソースからレポート生成
  3. Slack／メール送付
- **注意点**：
  - 実行スケジュール重複（多重起動）防止
  - トークン失効・認証切れの監視

---

## 🧠 まとめ

| 業務タイプ | 最適ツール | 理由 |
|--------------|--------------|------|
| 画面操作中心（承認・入力） | Playwright / UiPath | DOM操作・要素待機が安定、認識精度高い |
| ローカルアプリ中心（Excel, Notes等） | Power Automate Desktop / pywinauto | OSレベルで操作できる |
| システム間連携・通知中心 | Make / Zapier / n8n | APIで安定・ノーコード構築可能 |
| 大規模運用・監査必須 | UiPath / Automation Anywhere | 管理・配布・AI連携・監査対応 |

---

> **Tips:**
> - まずAPIの有無を調べ、「APIで完結できるか？」→「UI操作を自動化すべきか？」の順で判断。
> - PlaywrightやMakeを組み合わせる“ハイブリッドRPA構成”が2025年時点で主流。
